# 迁移指南

本文面向正在治理老流程系统，并考虑引入 Camunda 作为状态机的团队。

目标形态：

```text
Camunda 负责流程路由。
Open Flow Kernel 负责业务流程状态和生命周期。
```

## 概念映射

| 老系统概念 | Open Flow Kernel 概念 |
| --- | --- |
| 流程单 / 审批单 | `ProcessInstance` |
| 流程模板 / 流程类型 | `ProcessDefinition` |
| 审批节点 / 业务步骤 | `TaskDefinition` + `TaskInstance` |
| 节点处理代码 | `AbstractTask` / `TaskPhase` |
| 表单记录 | `FormInstance` + `FormService` |
| 表单提交 | `FormSubmitted` -> `TaskRelation` 完成 |
| 外部回调 / 签约 / 远程任务 | `ExternalTask` + `TaskRelation` |
| 子流程 | `SubProcessTask` + child `ProcessInstance` |
| 处理人 / 候选人 | `TaskCandidate` |
| 流程快照 / 可编辑过程数据 | `Packet` |
| 引擎变量 | 从流程数据投影出来的子集 |

## 迁移原则

1. 不要让 Camunda 成为业务数据库。Camunda 负责路由，内核负责业务状态。
2. 先拆清数据范围：流程数据、任务数据、表单数据、引擎变量、事件数据。
3. 优先迁移新流程，不要一开始迁移在途流程。
4. 老系统副作用放在 adapter 后面，不要拷贝内部 RPC、权限、通知、签约实现。
5. 每个状态迁移都需要可验证，可以用双写、对账、契约测试逐步切换。

## 推荐阶段

### 阶段 1：盘点

盘点每个流程类型和节点类型：

- 表单任务
- 人工审批任务
- 自动任务
- 外部回调任务
- 子流程任务
- 终止/取消路径

每个节点记录：

- 输入数据
- 输出数据
- 候选人规则
- 外部系统调用
- 重试规则
- 取消规则

### 阶段 2：建立内核模型

选择一个低风险流程：

1. 创建 `ProcessDefinition`。
2. 为每个节点创建 `TaskDefinition`。
3. 用 `AbstractTask` 或 `FormTask` 实现任务处理器。
4. 把老表单记录映射到 `FormService`。
5. 把外部等待映射到 `TaskRelation`。
6. 把候选人映射到 `TaskCandidate`。
7. 只有确实需要过程快照时才引入 `Packet`。

### 阶段 3：引入 Camunda 状态机

BPMN 中 task id 使用内核任务编码：

```text
BPMN userTask id = taskCode
Camunda businessKey = processInstanceId
```

BPMN 只负责路由，不承载业务审批事实。

### 阶段 4：新流程并行运行

新流程走新链路：

1. 通过 `ProcessService` 启动。
2. 保存内核流程/任务/表单状态。
3. Camunda 激活任务。
4. 通过内核表单/任务 API 完成。
5. 对比老系统最终输出和内核输出。

老流程继续留在老系统，直到新链路稳定。

### 阶段 5：切换

切换前必须确认：

- 每个任务都有稳定 `taskCode`。
- 老表单状态能映射到表单数据或任务数据。
- 外部回调能映射到 `TaskRelation`。
- 重复提交和重复回调是幂等的。
- 取消行为已经明确。
- 失败事件可以被发现和重试。

## 在途流程迁移

在途迁移风险更高，单独处理。

可选策略：

- 老流程继续在老系统完成。
- 从明确业务状态重新发起新流程。
- 重建内核流程/任务状态，并绑定到 Camunda execution。

第三种最难，需要映射：

- `ProcessInstance.status`
- 当前 `TaskInstance`
- 未完成 `TaskRelation`
- 表单提交状态
- Camunda active token

不要从这条路径开始。

## 示例映射

老流程：

```text
application order
-> submit application form
-> manager review form
-> finish
```

内核映射：

```text
ProcessDefinition: application-review
TaskDefinition: application
TaskDefinition: review
ApplicationFormTask extends FormTask
ReviewFormTask extends FormTask
FormService stores form data
TaskRelation waits for each form submission
```

运行时：

```text
ProcessService.start("application-review", {"applicant":"Alice"})
-> create ProcessInstance
-> start engine with businessKey = processInstanceId
-> engine creates task "application"
-> kernel creates TaskInstance
-> ApplicationFormTask creates form
-> form submit publishes FormSubmitted
-> TaskRelation completes
-> TaskInstance completes and advances engine
-> engine creates task "review"
-> review form repeats the same relation path
-> engine process end completes ProcessInstance
```
