# 持久化

本文记录当前 JDBC baseline、与参考实现的差异、以及后续持久化工作。

参考来源包括 `merchant-kernel-flow` 的迁移 SQL 和 repository 实现。

## 当前模块

```text
flow-kernel-persistence-jdbc
```

建表语句放在根目录：

```text
db/schema/h2/open-flow-kernel-jdbc-schema.sql
db/schema/mysql/open-flow-kernel-jdbc-schema.sql
```

`JdbcSchemaInitializer` 使用的是 H2 版本 schema，并由 `flow-kernel-persistence-jdbc` 在构建时从 `db/schema/h2` 打包进 jar。

字段级参考对照见 [参考 DDL 对照](参考DDL对照.md)。

当前提供：

- `JdbcProcessDefinitionRepository`
- `JdbcProcessInstanceRepository`
- `JdbcTaskInstanceRepository`
- `JdbcTaskRelationRepository`
- `JdbcPacketRepo`
- `JdbcPacketDataRepo`
- `JdbcTaskCandidateRepository`
- `JdbcEventStore`
- `JdbcCandidateRelationRepository`
- `JdbcProcessInstanceRelationRepository`
- `JdbcProcessInstanceDataInitRepository`
- `JdbcProcessLogRepository`
- `JdbcSchemaInitializer`

## 当前覆盖

已覆盖：

- 流程定义和任务配置。
- 流程实例创建、数据 merge、状态 CAS。
- 流程实例初始化配置表和流程实例 relation 表。
- 子流程 parent/task 关系字段。
- 任务实例创建、latest 查询、data merge、状态 CAS。
- 任务关系创建、查询、完成 CAS。
- task candidate 增删查、软删除、distinct by `ucid`。
- 原实现 `candidate_inst_relation` 待办关系表 DDL。
- 原实现 `candidate_inst_relation` 待办关系 repository baseline。
- packet pointer/value、commit CAS、expire reset、流程完成/取消状态更新。
- event record、listener delivery、retry due 查询、dispatch replay。
- `process_log` 运行日志表 DDL。
- `process_inst_relation`、`process_inst_data_init_config`、`process_log` repository baseline。

## 当前差异

| 项 | 当前状态 | 后续 |
| --- | --- | --- |
| DDL 方言 | H2/MySQL 两份 schema 已放入根目录 `db/schema` | MySQL/Testcontainers 运行需要本机 Docker daemon |
| Event V2 表 | 已有 JDBC baseline | 后续如需对齐原 JOOQ 表名/调度器再补 |
| `process_inst_relation` | DDL 和 JDBC repository baseline 已补齐 | 后续按管理端查询需要扩展复杂筛选 |
| `candidate_inst_relation` | DDL 和 JDBC repository baseline 已补齐 | full candidate source resolution 阶段再接业务来源 |
| `process_log` | DDL 和 JDBC repository baseline 已补齐 | 后续需要时补 batch insert 的完整时间字段控制 |
| `process_inst_data_init_config` | DDL 和 JDBC repository baseline 已补齐 | 后续需要时接入 ProcessService 初始化数据合并 |
| typed packet value | 已提供 typed `Packet<V>` API，JDBC 层仍以 JSON string 存储 value/history | 后续如需查询单个字段再补索引化扩展表 |

## 测试策略

repository 行为通过模块内 contract-style tests 校验。`flow-kernel-example` 和
`flow-kernel-persistence-jdbc` 分别跑同类断言，确保 example in-memory runtime 和 JDBC 行为一致。

当前接受命令：

```bash
./gradlew test --no-daemon --warning-mode all
```

## 第二轮 持久化原则

第二轮 不追求全量表 parity，只补 Camunda 最小闭环和 integrated example 必需的持久化能力。

全量 bounded parity 放到 第三轮。
