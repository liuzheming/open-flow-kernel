# 事件系统

事件系统不是普通通知组件。它承接流程生命周期执行、重试、失败和回放。

当前任务初始化和表单提交已经走事件路径。

## 核心语义

事件发布流程：

```text
publish event
-> 生成 eventId
-> 先记录事件
-> 事务提交后分发
-> 为每个 listener 创建 delivery
-> 执行 listener
-> 标记成功、重试或永久失败
```

## 事件信封

`EventEnvelope` 包含：

| 字段 | 用途 |
| --- | --- |
| `eventId` | 投递和消费幂等 |
| `payload` | 事件数据 |
| `occurredAt` | 发生时间 |
| `subjectType` / `subjectId` | 主体 |
| `partitionKey` | 顺序键，通常是流程实例 ID |
| `correlationId` | 请求/流程关联 |
| `causationEventId` | 触发来源事件 |

## 当前事件类型

- `ProcessEvent`
- `ProcessTaskEvent`
- `FormSubmitted`

JDBC baseline 已补齐事件记录、投递记录和 replay 所需的 record reconstruction。

## 分发模式

### 同步

用于必须在当前调用链内完成的监听器。

### After Commit

用于避免监听器读到未提交数据。当前通过 `TransactionBoundary` 抽象表达。

### 异步

用于可重试生命周期工作、远程调用、投影等。异步必须有明确失败记录。

## 重试

`RetryPolicy` 定义：

- 最大次数。
- 初始延迟。
- 倍率。
- 最大延迟。
- 可重试异常。
- 不可重试异常。

重试耗尽后进入永久失败状态。

## 当前实现

已实现：

- 事件先记录再分发。
- listener 注册。
- per-listener delivery。
- 同步/异步执行模式。
- retry/backoff。
- permanent failure。
- undispatched replay。
- retry due / retry one。
- JDBC event store。
- codec-based event payload reconstruction。

未完成：

- Spring transaction adapter。
- 生产级异步 executor。
- 面向运维的失败事件 API。

## JDBC 持久化

`JdbcEventStore` 位于：

```text
flow-kernel-persistence-jdbc/src/main/java/io/github/openflowkernel/jdbc/persistence/JdbcEventStore.java
```

它持久化两类数据：

- `event_record`：事件主体、状态、发生时间、subject、partition、correlation、causation。
- `event_delivery`：每个 listener 的投递状态、尝试次数、下次重试时间、最后错误。

payload 序列化通过 `JdbcEventPayloadCodec` 注入：

```text
DomainEvent -> String
String + payloadClass + eventType -> DomainEvent
```

这样 JDBC 模块不需要知道具体业务事件类，也不会把业务事件 JSON/反射策略硬编码进内核。

当前验证：

```text
JdbcEventStoreTest
-> record
-> dispatch
-> retry
-> delivery state
-> dispatchUndispatched replay after restart-like reconstruction
```
