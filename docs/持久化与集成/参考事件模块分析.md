# 参考事件模块分析

参考项目中的 flow 模块并不是只把事件当通知用。任务初始化等生命周期工作依赖事件模块执行，因此事件语义属于流程内核的一部分。

## 参考运行模型

```text
post event
-> 分配 eventId
-> 持久化事件记录
-> 如果存在本地事务，注册 after-commit 分发
-> 解析 listener
-> 为每个 listener 创建执行结果记录
-> 同步或异步执行
-> 记录成功、计划重试、永久失败
```

重试按 context key 分组，同一 context 内按 eventId 顺序执行，不同 context 可并行。

## 需要迁移的概念

| 参考概念 | open-flow-kernel 概念 |
| --- | --- |
| `EventV2.BasicInfo.eventId` | `EventEnvelope.eventId` |
| `eventGroup` | event type namespace |
| `contextKeyType/contextKey` | subject / partition key |
| `EventListenerV2` | `EventListener<T>` |
| `@Async` | `EventExecutionMode` |
| `@RetryPolicy` / `@Backoff` | `RetryPolicy` |
| event record | kernel event record |
| event result record | per-listener delivery |
| permanent failure table | failed delivery |
| retry schedule | due delivery retry |

## 参考限制

这些限制不直接照搬：

1. retry include/exclude 在参考实现中不完整。
2. executor reject 后可能丢弃任务。
3. 事件名注册依赖全局可变状态。
4. listener 发现依赖反射。
5. 事件数据对象可变。
6. 环境路由耦合内部部署概念。
7. 调度、持久化、执行、listener 发现集中在一个 manager。
8. 事务同步直接绑定 Spring。

## open-flow-kernel 拆分

```text
event core
  EventEnvelope
  EventListener
  RetryPolicy
  EventPublisher
  EventBus

event persistence
  EventStore
  EventDelivery

event execution
  EventExecutor
  TransactionBoundary

example
  InMemoryEventStore
  DirectEventExecutor
  ImmediateTransactionBoundary
```

第二轮 已支撑参考采用闭环。第三轮 已补 `JdbcEventStore` baseline，用 `JdbcEventPayloadCodec` 保持 payload 序列化边界；后续如需再对齐原 JOOQ 表名、Spring 事务适配和调度器。
