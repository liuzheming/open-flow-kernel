# 一致性与恢复

## 核心一致性问题

任务完成跨两个系统：

1. 内核持久化。
2. 工作流引擎。

当前复刻基线保留原版顺序：

```text
persist task COMPLETE
-> call WorkflowEngine.completeTask()
```

如果引擎调用失败，会出现内核任务已完成、引擎仍等待的窗口。这是原版顺序继承来的风险。

## 当前抽取规则

当前阶段不引入：

- `COMPLETING` 状态。
- command store。
- outbox 替代事件系统。
- reconciliation worker。

这些是后续改进候选，不是当前复刻目标。

## 后续改进候选

后续可以考虑：

```text
local transaction:
  persist task/process output
  ACTIVE -> COMPLETING
  insert CompleteEngineTask command

dispatcher:
  send command to engine

engine callback:
  COMPLETING -> COMPLETED
  postComplete
```

但这会改变原版状态模型，必须等复刻基线稳定后再讨论。

## 幂等要求

### 引擎命令

引擎适配器至少要安全处理：

- 重复 complete。
- 引擎任务已经完成。
- 引擎任务不存在但业务状态已到达目标状态。

### 事件消费

事件消费必须幂等：

- 如果目标状态已经到达，返回 no-op。
- 如果当前状态不兼容，记录冲突。
- 不能重复执行外部副作用。

## 恢复关注点

后续恢复工具可以扫描：

- 事件超过重试时间。
- relation 已完成但 task 仍 active。
- 引擎存在 active task 但内核没有对应 active task。
- 内核任务完成但引擎仍等待。

恢复动作必须显式、可审计。
