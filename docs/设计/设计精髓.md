# 设计精髓

`open-flow-kernel` 抽取的是一套流程内核设计，不是新的 BPMN 引擎，也不是通用审批平台。

一句话概括：

```text
用 Camunda 或其他流程引擎负责状态机和路由；
用内核自己的 Process、Task、Form、Relation、Event、Packet、Candidate 承载业务流程语义。
```

## 内核负责什么

内核负责业务事实：

- `ProcessInstance`：业务流程实例、流程状态、父子流程关系、流程级数据。
- `TaskInstance`：业务任务实例、任务状态、生命周期执行、任务级数据。
- `TaskRelation`：任务等待外部实体完成的关系。
- `FormService`：表单创建、提交、读取的适配边界。
- `EventPublisher` / `EventStore`：生命周期事件、重试、失败和回放。
- `Packet`：流程数据快照和当前指针。
- `TaskCandidate`：任务候选人/处理人状态。

流程引擎负责路由：

- 当前激活哪个 BPMN 节点。
- 何时创建引擎任务。
- 何时结束流程。
- 网关和 token 流转。

核心身份映射只有两条：

```text
engine businessKey = business processInstanceId
engine taskDefinitionKey = business taskCode
```

## 看代码时先看什么

先看 `ProcessService`。

它先创建业务流程实例，再启动流程引擎。这说明业务流程实例才是业务事实源，引擎只是适配器。

再看 `WorkflowEventListener`。

它把引擎回调转换成内核动作：创建业务任务、完成业务任务后置动作、取消任务、结束业务流程。

再看 `TaskFactory` 和 `TaskInstance`。

`TaskFactory` 根据任务配置创建任务处理器。`TaskInstance` 负责生命周期顺序：

```text
create -> init -> afterInit -> wait relations -> beforeComplete
-> persist output -> complete engine task -> postComplete
```

任务处理器只实现业务行为，状态流转和引擎推进由 `TaskInstance` 控制。

再看 `TaskRelationService`。

表单、子流程、外部回调都走同一个规则：

```text
外部实体完成
-> relation 完成
-> 检查任务所有 relation
-> 全部完成后任务只完成一次
```

这就是设计的关键：不同外部系统不再各自发明一套任务完成机制。

最后看 Event、Packet、Candidate。

- Event 不是普通通知，而是生命周期执行和重试机制。
- Packet 是挂在流程生命周期上的数据快照能力。
- Candidate 是挂在任务生命周期上的候选人能力。

## 不要误解

- 不要把这个项目理解成 Camunda 替代品。
- 不要把表单理解成流程本身，表单只是任务等待的一类外部关系。
- 不要把 `flow-kernel-example` 里的 in-memory 代码当成生产适配器。
- 不要把当前直接调用引擎完成任务的方式理解成最终可靠性方案；它只是当前复刻阶段保留的原版顺序。
