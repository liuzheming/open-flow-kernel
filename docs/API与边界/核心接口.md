# 核心接口

本文列出使用者和适配器作者最需要理解的接口。

## 接口分组

| 分组 | 主要类型 | 责任方 |
| --- | --- | --- |
| 应用入口 | `ProcessService`, `ITask` | 业务应用 |
| 任务扩展 | `AbstractTask`, `TaskPhase`, `TaskResult`, `TaskContext` | 流程使用者 |
| 引擎适配 | `WorkflowEngine`, `WorkflowEngineListener` | 基础设施适配器 |
| 表单适配 | `FormService`, `FormSubmissionListener` | 表单系统 |
| 关系内核 | `TaskRelationRepository`, `TaskRelationService` | 内核 / 持久化适配器 |
| 事件内核 | `EventPublisher`, `EventListener`, `EventStore`, `EventBus` | 内核 / 事件适配器 |
| Packet baseline | `PacketService`, `PacketRepo`, `PacketDataRepo` | 内核 / 持久化适配器 |
| Candidate baseline | `TaskCandidateService`, `TaskCandidateRepository` | 内核 / 持久化适配器 |

## `ProcessService`

包：`io.github.openflowkernel.core.process`

流程生命周期入口：

```java
long start(String definitionKey, Map<String, String> initialData);
void complete(long processInstanceId);
void suspend(long processInstanceId);
void processContinue(long processInstanceId);
void cancel(long processInstanceId);
ProcessInstance get(long processInstanceId);
```

关键规则：

- `start` 先创建业务流程实例，再启动引擎。
- 引擎 `businessKey` 使用业务 `processInstanceId`。
- `complete` 通常由引擎 process-end 回调触发。
- 状态变更使用 repository CAS。

## 任务扩展接口

### `ITask`

业务任务句柄：

```java
long getProcessInstanceId();
long getTaskInstanceId();
String getTaskCode();
ProcTaskStatusEnum getStatus();
void complete(TaskResult result);
void cancel();
```

### `AbstractTask`

任务处理器基类。业务任务通常继承它，然后覆盖 `TaskPhase` 生命周期方法。

### `TaskPhase`

```java
TaskResult init(TaskContext context);
void afterInit(TaskContext context);
TaskResult beforeComplete(TaskContext context);
TaskResult postComplete(TaskContext context);
```

生命周期顺序：

```text
engine task-created
-> create business task
-> init
-> afterInit
-> wait relations
-> beforeComplete
-> persist output
-> complete engine task
-> postComplete
```

`TaskInstance` 控制状态流转和引擎推进，任务处理器只实现业务行为。

### `TaskContext`

携带：

- 流程实例 ID
- 任务实例 ID
- 流程数据
- 当前任务数据
- 任务配置
- 任务编码

### `TaskResult`

生命周期方法和显式完成任务时返回的数据变更：

- task data updates
- process data updates

## 引擎接口

### `WorkflowEngine`

包：`io.github.openflowkernel.core.engine`

```java
void setListener(WorkflowEngineListener listener);
void start(String processDefinitionKey, String businessKey, Map<String, Object> variables);
void completeTask(String processDefinitionKey, String businessKey, String taskCode, Map<String, Object> variables);
```

身份映射：

```text
businessKey = business processInstanceId
taskCode = engine taskDefinitionKey
```

### `WorkflowEngineListener`

```java
void onTaskCreated(EngineTaskCreated event);
void onTaskCompleted(EngineTaskCompleted event);
void onTaskDeleted(EngineTaskDeleted event);
void onProcessCompleted(EngineProcessCompleted event);
```

## 表单接口

模块：`flow-kernel-form`

包：`io.github.openflowkernel.form`

### `FormService`

```java
void setSubmissionListener(FormSubmissionListener listener);
long create(String definitionKey, Map<String, String> initialData);
FormInstance get(long formInstanceId);
void submit(long formInstanceId, Map<String, String> submittedData);
void cancel(long formInstanceId);
```

表单提交不会直接完成任务，而是进入：

```text
FormSubmitted -> TaskRelation complete -> Task complete
```

### `FormTask`

表单任务基类，负责：

- 初始化表单。
- 注册 `RELATION_FORM`。
- 保存 `FORM_INST_ID`。
- 存在 `FORM_DEF_MAP` 时创建多个表单，保存 `FORM_INST_MAP`。
- 支持 `FORM_INPUT_MAP` / `FORM_OUTPUT_MAP` 的多表单输入输出映射。
- 在 `beforeComplete` 中读取提交值并映射到流程数据。

## Relation 接口

### `TaskRelationRepository`

```java
TaskRelation create(long taskInstanceId, String relationType, String relationInstanceId);
Optional<TaskRelation> find(String relationType, String relationInstanceId);
List<TaskRelation> findByTaskInstanceId(long taskInstanceId);
boolean compareAndSetCompleted(long relationId);
```

通用规则：

```text
外部实体完成
-> relation 完成
-> 检查任务所有 relation
-> 全部完成后任务只完成一次
```

## Event 接口

### `EventPublisher`

```java
<T extends DomainEvent> EventEnvelope<T> publish(EventDraft<T> event);
```

### `EventListener<T>`

```java
void listen(EventEnvelope<T> event);
```

### `EventStore`

事件存储负责：

- 先记录 event。
- 记录每个 listener 的 delivery。
- 标记成功、重试、永久失败。
- 支持未派发事件回放和单 listener 重试。

## Packet baseline

`PacketService` 当前保留了指针和值历史语义：

- 初始化 packet。
- 获取 current/latest。
- commit 时写入新 value，并用 pointer CAS 防并发覆盖。
- expire 时重置当前指针。
- 流程完成/取消时更新 packet 状态。
- typed `Packet<V>` 子类外壳。
- `PacketFactory` 通过注册的 resolver class 创建 typed packet。
- `PacketInitManager` 根据 `PacketInitConfig.name` 调用 `ProcessPacketInit` 初始化 packet。
- `ProcessService.start` 通过 `ProcessStartHook` 扩展点接入 `PacketProcessStartHook`，在
  workflow 启动前初始化 packet。

仍未复刻：

- 业务 packet 子类、ES/合同/签约输出 helper。
- `noResetPacketProcInstIds` 运行时运维覆盖。
- parent/root packet commit helper。

## Candidate baseline

`TaskCandidateService` 当前保留：

- 按 `ucid` 去重。
- 新候选人插入。
- 不再存在的候选人软删除。
- 通过 `TaskCandidateSource#select(selectionConfig, processInstData)` 解析候选人来源。
- `TaskInstance.init()` 在任务初始化后读取 `CANDIDATE_SELECTION` 并触发候选人刷新。

这对应原实现的 `TaskInstance.initCandidate -> CandidateSelector.refresh -> select -> diffAndUpdateCandidates` 主链路。

仍未复刻：

- Groovy `CANDIDATE_SELECTION` / `CandidateScriptBase`。
- role/personnel/auth based candidate source resolution。
- external missing-candidate notification。
- candidate change event side effects for search/todo sync。
- `candidate_inst_relation` 的复杂待办节点查询。

`CandidateRelationRepository` 对应原实现 `CandidateRepo` / `candidate_inst_relation`：

- `addCandidates` 增加 action/task relation 候选人。
- `list` / `listEntities` 查询未删除候选人。
- `updateCandidateStatus` 更新候选关系状态。
- `updateCandidateDeleted` / `deleteCandidates` 软删除。
- `distinctCandidateByActionItemNos` 按 `ucid` 去重。

完整候选人来源解析、待办节点复杂查询和 personnel/role/Groovy 集成仍属于 deferred。

## Generic Persistence Repositories

以下接口用于表达原实现的通用表边界：

- `ProcessInstanceRelationRepository`：对应 `process_inst_relation`，提供流程实例外部 relation 的新增和查询。
- `ProcessLogRepository`：对应 `process_log`，提供流程/任务日志写入和按流程查询，content 按原实现截断到 2000 字符。
- `ProcessInstanceDataInitRepository`：对应 `process_inst_data_init_config`，提供流程定义 + 城市维度初始化数据的 init/get/update/delete。

这些接口当前有 JDBC baseline；复杂管理端查询、Redis lock、自动合并到 `ProcessService.start` 仍不在核心闭环内。

## SubProcess baseline

`SubProcessTask` 当前支持：

- `SUB_PROCESS_DEF_CODE` 单子流程。
- `SUB_PROCESS_DEF_CODE_MAP` 多子流程。
- `FORM_INPUT` 初始化子流程输入。
- 单子流程 `FORM_OUTPUT` 输出映射。
- 子流程完成后通过 `subProcessTaskType` relation 推动父任务。

业务分支、candidate 传播、packet init、复杂 output map 属于后续对齐项。
