# 模块边界

本文说明 `open-flow-kernel` 各模块的职责、依赖方向和 public 可见性的含义。

当前项目处于高保真抽取 baseline 阶段。为了让纯 Java 装配、测试和示例足够直接，部分 runtime wiring 类和 adapter 实现仍然是 Java `public`。这不等于它们都是稳定业务 API。

## 模块职责

| 模块 | 职责 | 读者应该如何使用 |
| --- | --- | --- |
| `flow-kernel-core` | Process、Task、Relation、Candidate、WorkflowEngine SPI 和核心生命周期 | 业务系统主要依赖的核心模块 |
| `flow-kernel-event` | EventV2、EventBus、EventStore、listener delivery、retry/replay | 事件基础语义和事件 adapter SPI |
| `flow-kernel-form` | FormService、FormTask、表单提交到 TaskRelation 的桥接 | 接入真实表单系统或实现表单任务 |
| `flow-kernel-packet` | Packet pointer/value/history、typed Packet、packet init hook | 流程数据包扩展 |
| `flow-kernel-engine-camunda` | Camunda 7 adapter baseline | 使用 Camunda 作为状态机时复用或替换 |
| `flow-kernel-persistence-jdbc` | JDBC repository adapter、schema initializer、event JDBC store | 持久化 adapter 示例和可复用 baseline |
| `flow-kernel-example` | in-memory reference runtime 和完整示例 | 只用于理解设计、跑测试、写迁移 spike，不是生产 adapter |

## 依赖方向

当前依赖方向刻意保持简单：

```text
flow-kernel-core -> flow-kernel-event
flow-kernel-form -> flow-kernel-core + flow-kernel-event
flow-kernel-packet -> flow-kernel-core
flow-kernel-engine-camunda -> flow-kernel-core
flow-kernel-persistence-jdbc -> flow-kernel-core + flow-kernel-event + flow-kernel-packet
flow-kernel-example -> flow-kernel-core + flow-kernel-event + flow-kernel-form + flow-kernel-packet
```

关键约束：

- `flow-kernel-core` 不依赖 Camunda、JDBC、Spring、表单平台或 packet 模块。
- Camunda 只通过 `WorkflowEngine` / `WorkflowEngineListener` 接入核心。
- Form 通过 `FormService` 和 `FormSubmitted` 事件接入核心，不把 UI/form-platform 细节放进 core。
- Packet 通过 `ProcessStartHook` 接入 `ProcessService.start`，避免 core 反向依赖 packet。
- JDBC 是 adapter，不是 domain model 的来源。
- Example 是 reference runtime，不是 adapter SPI 的新增层。

## Public 可见性分层

Java `public` 类型分成四层：

| 层级 | 类型例子 | 稳定性 |
| --- | --- | --- |
| 用户 API | `ProcessService`, `AbstractTask`, `FormTask`, `SubProcessTask`, `PacketService`, `TaskCandidate` | 当前 baseline 的主要使用面 |
| Adapter SPI | `WorkflowEngine`, `FormService`, `EventStore`, repository interfaces, `TaskCandidateSource`, `ProcessPacketInit` | 允许使用者实现自己的基础设施适配 |
| Adapter implementation | `CamundaWorkflowEngine`, `JdbcProcessInstanceRepository`, `JdbcEventStore` | 可复用，但跟对应基础设施绑定 |
| Runtime wiring | `WorkflowEventListener`, `TaskInitializationEventListener`, `FormSubmittedEventListener`, Camunda listener classes | public 只是为了装配，不应被业务代码直接依赖 |

更细的接口列表见 [公共 API 边界](公共API边界.md) 和 [核心接口](核心接口.md)。

每个主要模块目录下也有简短 `README.md`，用于从 GitHub 文件树直接进入对应模块边界。

## 不应该跨越的边界

- 业务任务不要直接操作 Camunda task。任务完成必须回到 `TaskInstance.complete` / `WorkflowEngine.completeTask` 的顺序。
- 表单提交不要直接 complete engine task。它应该先完成 `TaskRelation`。
- Candidate 来源解析不要写进 `TaskInstance`。应该实现 `TaskCandidateSource`。
- Packet 初始化不要写进 `ProcessService`。应该通过 `ProcessStartHook` / `PacketProcessStartHook`。
- JDBC repository 不应该引入业务平台字段含义。字段 parity 见 [参考 DDL 对照](../持久化与集成/参考DDL对照.md)。

这些边界来自原 `merchant-kernel-flow` 的核心设计：Camunda 是状态机，业务流程事实由 kernel 持有。
