# 项目计划

本文是 `open-flow-kernel` 的长期计划锚点，用于防止目标在长对话或上下文压缩后丢失。

## 原始目标

创建一个独立开源项目，基于参考 `merchant-kernel-flow` 的核心设计和实现，抽取出一套可阅读、可运行、可借鉴的流程内核。

必须尽可能忠于原设计：

1. Camunda 只作为状态机和流程路由引擎。
2. 自研 `Process -> Task -> Form` 三层流程模型仍是业务语义核心。
3. 保留关键类职责、生命周期顺序、事件方向、数据边界和持久化语义。
4. 不为了“看起来更现代”而重写成另一套框架。
5. 只有在原实现明显不适合开源、依赖内部基础设施、存在法律风险或无法独立运行时，才做有意偏离。
6. 所有有意偏离必须在文档中说明。

这是高保真抽取，不是全新流程框架。

## 工作约定

### 参考范围

`merchant-kernel-flow` 是抽取根。若其核心流程语义依赖其他模块，则迁移必要的通用闭包，例如：

- event 发布、订阅、retry、异步分发和 event metadata；
- form 公共契约和生命周期；
- packet / snapshot 行为；
- 流程公开行为需要的通用异常或结果契约。

不迁移内部 RPC、权限、签约、通知、人员系统等业务平台实现。

### 新项目范围

所有新实现都在独立仓库 `open-flow-kernel` 中完成，不作为商业仓库子目录或子模块。

### Agent 使用

当前阶段不需要为了使用而使用多 agent。除非后续有明确并行任务，例如“独立审计 DDL”或“独立梳理事件模块”，否则由当前主 agent 持续推进即可。

## 第二轮：Camunda + Process-Task-Form 闭环

目标：让新读者可以运行一个完整示例，并理解 Camunda 只做状态机、业务事实在内核中。

已完成：

- `flow-kernel-engine-camunda` adapter。
- Camunda in-memory 两用户任务 BPMN 闭环。
- `ProcessService -> FormTask -> TaskRelation -> CamundaWorkflowEngine` 集成测试。
- 完整示例文档。
- API 边界、模块边界、迁移指南。

完成标准：

- 一条命令跑通默认测试。
- Camunda 7 adapter 通过最小闭环。
- 迁移指南足够具体。
- README 指向示例、Camunda 映射、API 边界和迁移文档。
- 示例行为能映射回参考设计。

## 第三轮：有界核心对齐和开源交付

目标：在不无限扩大范围的前提下，把核心复刻做到可读、可运行、可作为老流程系统治理参考。

已完成：

- Gradle 多模块和 CI baseline。
- 中文 docs 分类和文档导航。
- typed `Packet<V>`、`PacketFactory`、`PacketInitManager`、process-start packet init。
- `TaskInstance.initCandidate`、`TaskCandidateSource`、`candidate_inst_relation` baseline。
- multi-form 通用配置。
- multi-subprocess 通用配置。
- Event JDBC persistence 和 replay。
- `process_inst_relation`、`process_log`、`process_inst_data_init_config` repository baseline。
- H2/MySQL DDL。
- schema parity test。

不进入 第三轮：

- Maven Central 发布执行。
- 生产级 Spring Boot starter。
- UI / admin console。
- 完整 observability stack。
- benchmark。
- 业务审批平台全量行为。
- 内部人员、权限、签约、RPC 实现。
- 参考核心不存在的新流程概念。

## 当前仍需外部确认

这些不阻塞核心复刻闭环，但影响“正式公开发布”或“真实环境兼容性声明”：

- `LICENSE` 仍处于法务确认状态。
- `SECURITY.md` 仍需正式公开安全联系人。
- Docker 可用后可运行 `mysqlIntegrationTest` 证明真实 MySQL compatibility。

## 下一步优先级

1. 保持文档组织清晰，避免新读者在审计文档里迷路。
2. 保持默认测试和 schema parity test 通过。
3. 如继续补能力，必须先确认它属于参考核心，而不是后续增强。
4. 如准备开源发布，先解决 license/security contact，再做 CI 首次真实验证。
