# 第二轮执行计划

第二轮的目标不是继续堆功能，而是让一个第三方流程系统负责人能用 `open-flow-kernel` 理解这套方案，并判断能否用于自己的老流程系统治理。

核心判断标准：

1. 能看懂：Camunda 为什么只作为状态机。
2. 能跑通：`Process -> Task -> Form/Relation -> Engine` 有最小闭环。
3. 能映射：老系统的流程、任务、表单、回调、数据变更能映射到内核概念。
4. 能扩展：知道哪些接口是用户 API，哪些是 adapter SPI，哪些只是内部 runtime wiring。

## 阶段 1：公共 API 边界

目标：降低读者第一次进入代码库时的理解成本。

交付内容：

- 明确 `ProcessService`、`AbstractTask`、`FormTask`、`SubProcessTask`、`WorkflowEngine`、`FormService`、`EventStore` 的角色。
- 区分用户 API、adapter SPI、内部 runtime wiring。
- 在文档中说明 example in-memory 代码只是示例实现，不是生产 adapter。
- 对包结构和模块命名做一次复盘，避免 memory/example/test contract 这类名称误导读者。

完成标准：

- `docs/API与边界/公共API边界.md` 能回答“我应该实现什么、调用什么、不要依赖什么”。
- `docs/README.md` 的阅读路径能把读者从设计导到接口，再导到示例。

## 阶段 2：完整示例

目标：用一个完整流程把核心设计串起来，而不是只让读者看零散单测。

交付内容：

- 保留一个 `flow-kernel-example` 模块作为唯一示例入口。
- 示例覆盖流程启动、任务创建、任务初始化、表单创建、表单提交、relation 完成、任务完成、流程推进。
- 示例中使用 Event、Form、TaskRelation、Packet 或 Candidate 的地方必须解释为什么这么挂接。
- 示例代码要尽量忠于原版设计，不引入新的抽象风格。

完成标准：

- 读者可以从 `docs/设计/完整示例.md` 直接定位到完整测试和核心类。
- 示例能用 `./gradlew test --no-daemon --warning-mode all` 跑通。

## 阶段 3：Camunda 最小闭环

目标：证明“Camunda 只作为状态机”不是一句口号，而是有清晰 adapter 边界。

交付内容：

- 定义 Camunda 7 adapter 的最小职责：deploy/start process、complete task、接收 task-created / process-ended 回调。
- 明确身份映射：`businessKey -> ProcessInstance.id`，task definition key -> `TaskDefinition.code`。
- 明确内核与 Camunda 的职责边界：业务状态、任务数据、表单关系、packet/candidate 不落入 Camunda。
- 提供 `flow-kernel-engine-camunda` 最小 adapter baseline。
- 用真实 Camunda in-memory engine 和 BPMN 示例补齐可执行闭环测试。

完成标准：

- `docs/设计/Camunda作为状态机.md` 能让读者理解 Camunda 在这套设计里的最小位置。
- 后续真正接 Camunda 时，不需要重新设计核心接口。

当前状态：

- 已完成 `flow-kernel-engine-camunda` adapter baseline。
- 已完成 `CamundaWorkflowEngineTest` 可执行闭环：deploy BPMN、start、task create、task complete、process end。
- 已完成 `CamundaProcessTaskFormIntegrationTest`，覆盖完整
  `ProcessService -> FormTask -> TaskRelation -> Camunda` 集成链路。

## 阶段 4：迁移指南

目标：让有老流程系统治理压力的读者能做概念映射和分阶段迁移。

交付内容：

- 旧系统概念到新模型的映射表。
- 从只读镜像、旁路运行、单流程试点、增量切换到补齐可靠性的迁移步骤。
- 明确哪些能力必须先迁移，哪些能力可以后置。
- 用一个典型审批流程说明如何拆成 process、task、form、relation、event。

完成标准：

- `docs/迁移与治理/迁移指南.md` 能回答“我如何把自己的老流程系统迁到这个模型”。
- 文档不承诺当前代码尚未实现的生产可靠性能力。

## 阶段 5：设计证明文档

目标：把当前实现和原版设计的关键一致性说明清楚。

交付内容：

- 状态机文档说明 process/task/form/relation 的状态变化。
- 运行时序文档说明启动、初始化、提交、完成、事件分发顺序。
- 数据模型文档说明主要实体和关系。
- 一致性与恢复文档说明当前高保真复刻的边界，以及后续可改进点。

完成标准：

- 读者不需要读完整源码，也能掌握核心生命周期。
- 文档明确区分“已复刻”“示例实现”“后续增强”。

## 不在第二轮范围内

以下内容在第二轮时不阻塞闭环；其中有界核心部分已经在第三轮补齐，生产化能力仍后置：

- typed packet 生命周期 baseline 已在第三轮补齐；业务 packet 子类和平台 helper 后置。
- candidate source SPI 和 task-init wiring 已在第三轮补齐；完整人员/角色/Groovy 来源解析后置。
- multi-form / multi-subprocess 通用配置 baseline 已在第三轮补齐；业务 UI/平台 side effects 后置。
- Event JDBC persistence baseline 已在第三轮补齐；Spring scheduler/transaction adapter 后置。
- MySQL/Testcontainers 显式任务已存在并可编译；真实运行仍依赖 Docker daemon。
- 生产级 command delivery、reconciliation、幂等补偿。

这些内容进入第三轮或后续生产化增强，前提是不能改变当前从原版设计抽取出的核心模型。
