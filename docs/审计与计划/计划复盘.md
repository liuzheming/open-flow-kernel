# 计划复盘

## 复盘视角

假设读者负责一批老流程系统，想引入 Camunda 作为状态机，同时保留业务流程语义在业务系统内。

从这个视角看，项目必须快速回答：

1. Camunda 如何只做路由状态机？
2. `Process -> Task -> Form/Relation` 如何跑完整链路？
3. Event、Packet、Candidate、SubProcess 如何挂到生命周期上？
4. 哪些接口应该实现，哪些类只是内部装配？
5. 老系统应该如何迁移到这个模型？

## 当前判断

当前代码已经证明设计方向，但下一步最有价值的不是继续补齐所有参考功能，而是让第三方流程负责人能看懂、能跑通、能映射。

因此 第二轮 定义为“参考采用闭环”：

- Camunda 7 最小闭环。
- 一个完整 integrated example。
- 迁移指南。
- public/internal API 边界。
- 状态、数据、事件 walkthrough。

第三轮 的核心有界对齐当前状态：

- typed packet lifecycle 已补。
- `TaskInstance.initCandidate` 和 candidate source SPI baseline 已补；完整 personnel/role/Groovy source resolution 明确 deferred。
- multi-form 和 multi-subprocess 通用配置已补。
- Event V2 JDBC persistence 已补。
- 缺失通用持久化表 DDL 和 JDBC repository baseline 已补。
- MySQL/Testcontainers 仍待验证。

## 决策

不要让完整 Packet、Candidate、Form、SubProcess、Event parity 阻塞 Camunda 闭环和迁移文档，除非缺失部分是 integrated example 必须使用的。

项目应该先成为“可用于评估和借鉴的参考设计”，然后再补齐 bounded parity。
