# 设计来源与偏离

本文记录 `open-flow-kernel` 与参考实现的对应关系。它的目的不是做新设计说明，而是回答：

- 哪些行为来自参考项目。
- 哪些地方为了开源、独立运行或基础设施解耦做了有意偏离。
- 哪些能力属于后续范围，不能混进当前核心复刻。

## 抽取原则

参考实现是原 `merchant-kernel-flow` 模块，以及为保持流程语义所必须依赖的通用模块，例如 event、form、packet。

当前阶段是高保真抽取，不是重新设计：

1. 保留 `Process -> Task -> Form` 三层模型。
2. 保留 Camunda 只作为状态机和路由引擎的定位。
3. 保留关键生命周期顺序、事件方向、状态含义、配置键和持久化语义。
4. 内部基础设施只做薄 adapter 替换，不把商业系统依赖带入开源项目。
5. 有意偏离必须记录原因。

## 核心映射

| 当前组件 | 参考来源 | 已保留行为 | 有意偏离 | 原因 |
| --- | --- | --- | --- | --- |
| `ProcessService` | `flow/ProcessService` | 先创建业务流程实例，再启动流程引擎 | 纯 Java 依赖 | 保持 core 不依赖 Spring 和内部服务 |
| `WorkflowEventListener` | `flow/listener/WorkflowEventListener` | 引擎任务事件创建/完成业务任务 | 引擎中立事件 API | Camunda 类型不能进入 core |
| `TaskFactory` | `flow/core/TaskFactory` | 按配置创建任务；同流程同任务码下处理 latest task；完成任务通过 `LAST_TASK_INST_ID` 串联 | 示例内存 adapter 使用进程内同步，不迁移 Redisson | 替换内部基础设施，但保留语义 |
| `TaskInstance` | `flow/core/TaskInstance` | 负责 init、afterInit、beforeComplete、complete、postComplete、数据 merge 和引擎推进 | 显式依赖注入 | 去除容器查找和隐藏耦合 |
| `TaskPhase` | `flow/TaskPhase` | `init`、`afterInit`、`beforeComplete`、`postComplete` 生命周期 | 当前只保留 `TaskContext`/`TaskResult` | 先保留生命周期，不引入多余泛型复杂度 |
| `TaskRelationService` | `flow/service/TaskRelationService` | 所有外部 relation 完成后任务才继续 | 乐观更新和重复回调幂等 | 让外部回调幂等语义显式化 |
| `ExternalTask` | `flow/task/ExternalTask` | 注册外部 relation，等待全部 relation 完成 | 构造器注入 | 保留扩展点，去除 Spring 字段注入 |
| `FormTask` | `flow/task/FormTask` | 单表单和通用多表单配置；表单提交先完成 `TaskRelation`，再完成任务 | 独立 `flow-kernel-form` adapter SPI；不迁移 UI/菜单/资源清理副作用 | 让表单平台可替换 |
| `SubProcessTask` | `flow/task/SubProcessTask` | 支持单子流程和通用多子流程配置；子流程完成后回写 relation | 业务分支和多子流程聚合输出后置 | 当前只复刻通用子流程生命周期 |
| `flow-kernel-engine-camunda` | `WorkflowCenterProxy`、`PublishDelegateParseListener`、`WorkflowEventListener` | 通过 process key/business key/task definition key 启动和完成；parse listener 注入 Camunda listener | 普通 Camunda adapter，不做 Spring Boot starter | 保留 Camunda-as-state-machine，不泄漏 Camunda 到 core |
| `EventV2` / `EventBus` | `merchant-kernel-event` | 事件先记录、事务后分发、listener delivery、retry、replay、永久失败 | 调度器、事务管理、JOOQ record 后置 | 保留事件语义，基础设施 adapter 化 |
| `Packet` | `packet/PacketFacadeImpl`、`PacketRepo`、`PacketDataRepo` | pointer/value/history、commit CAS、expire reset、typed packet、init manager、流程启动初始化 | 业务 packet 子类、签约/合同/ES helper 后置 | 保留版本化数据语义，排除业务专属实现 |
| `TaskCandidate` | `TaskCandidateRepo`、`CandidateSelector` | 候选人增删查、软删除、distinct by `ucid`、diff update、任务初始化 SPI | Groovy、角色/人员/权限源、通知、搜索待办后置 | 保留任务生命周期中的候选人语义，排除内部人员系统 |
| JDBC repositories | `ProcessRepoImpl`、`TaskRepoImpl`、packet repo、迁移 SQL | process/task/relation/packet/candidate/event 的核心持久化语义和 DDL baseline | 普通 JDBC + schema initializer，不迁移 JOOQ/Spring transaction | 保持独立可运行 |

## 依赖模块处理

| 参考依赖 | 处理方式 | 范围 |
| --- | --- | --- |
| `merchant-kernel-event` | 迁移通用核心，持久化和调度 adapter 化 | 事件 envelope、listener registry、retry、delivery、replay |
| `merchant-kernel-form` | 迁移公共契约和生命周期 | 保持 `FormTask` 行为，表单平台外置 |
| `merchant-kernel-common` | 选择性迁移 | 只保留流程公开行为需要的通用契约 |
| `merchant-kernel-rpc` | 不迁移实现 | 内部外部系统集成不是流程内核 |
| 业务平台/人员/权限/签约模块 | 不迁移实现 | 通过 SPI 或 deferred scope 记录 |

## 后置能力

这些能力不是被否定，而是不属于当前有界核心复刻：

- 完整候选人来源解析和人员/角色/权限系统。
- 业务 UI、表单菜单、表单资源清理和平台 relation 副作用。
- 审批平台 action/audit/task retry UI。
- 业务专属 packet 子类和外部系统 helper。
- Spring Boot starter、生产监控、发布流水线、Maven Central 发布。
- durable command store、reconciliation worker、`COMPLETING` 等参考实现没有的改进设计。

集中边界见 [延后范围](../迁移与治理/延后范围.md)。

## 事件模块抽取原则

必须保留：

- 分发前先持久化事件。
- 一个事件一个稳定 ID。
- 事务提交后再分发。
- 每个 listener 独立 delivery 状态。
- 支持同步/异步 listener。
- 支持 retry、next retry time、attempt count。
- 支持 exhausted/permanent failure。
- 支持未分发事件 replay 和失败 listener 手动重试。

不进入当前复刻：

- JOOQ 生成 record。
- 内部调度框架注解。
- 私有链路追踪注解。
- 内部环境枚举、实体枚举。
- Redis 和具体 EventBus 实现细节。
